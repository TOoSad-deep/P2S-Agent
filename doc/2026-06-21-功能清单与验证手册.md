# 第 0 部分 · 前置与启动

## 0.1 启动顺序
1. 配置 `backend/.env`：至少填一个模型槽位的 `MODEL_1_API_KEY` / `MODEL_1_BASE_URL` / `MODEL_1_MODEL`（详见第一部分「环境变量配置」）。没有 `.env` 时 `start.sh` 会从 `.env.example` 复制一份。
2. 在仓库根目录运行 `./start.sh`（或 `./start.sh restart`）：自动清理旧进程、启动后端(8001) + 前端(5174)。
3. 健康检查：`curl http://localhost:8001/health` → 预期 `{"status":"ok","agent":"p2s-agent"}`。
4. 浏览器打开 `http://localhost:5174`。

## 0.1.1 已知环境坑（务必先看，否则会误判为 bug）
- **Vite 端口自增**：5174 被占时 Vite 会**静默漂移**到 5175/5176…，而后端渲染服务写死连 `FRONTEND_URL`(默认 5174)，导致截图阶段报 `net::ERR_CONNECTION_REFUSED`。务必保证**只有一个**前端实例独占 5174（先 `./start.sh clean` 再起）；可在 `frontend/vite.config.ts` 加 `strictPort:true` 让冲突直接报错。
- **重复进程**：反复手动启动会留下多个 uvicorn/vite 孤儿进程抢端口。统一用 `./start.sh clean` / `restart`。
- **macOS 文档目录访问 (TCC)**：数据根 `backend/test_results/` 位于 `~/Documents` 下（受 macOS 隐私保护）。若后端以 `uvicorn --reload` 启动且启动方（终端/IDE）未获「完全磁盘访问权限」，reload 出来的 worker 子进程会对该目录报 `PermissionError [Errno 1]`。代码已做读路径降级（不再 500，API 仍可用），但要真正**落盘持久化**，需给启动方授予「系统设置 → 隐私与安全性 → 完全磁盘访问权限」，或把数据根移出 `~/Documents`。

## 0.2 端到端冒烟验证（约 5 分钟，串起主要功能）
1. **生成**：Studio 页拖入一张 PNG → 选模型预设 + 策略预设 → 点「生成」。预期顶部状态变 `running`，候选评分板与时间线开始填充。
2. **完成**：等到状态 `completed` → 「已选候选」出现 GLSL，「图像对比」可见参考图 vs 着色器实时渲染。
3. **分支**：切到 Canvas 页 → 该 run 作为根节点出现 → 选一个检查点「从此处定向优化」→ 填自然语言反馈 → 生成子分支 run。
4. **比较**：子分支完成后，用「对比条」比较父/子两版。
5. **进阶（可选）**：发起「变体探索」(一对多) / 「批量抽卡」/ 「融合」，验证扇出与合成目标。
- **通过标准**：每步状态都能 `running → completed`；产物可下载；前后端控制台无红色报错。

### 一、服务启动与生命周期（start.sh）

#### 功能：一键启动后端 + 前端
- **功能**: 同时拉起 FastAPI 后端（8001）和 Vite 前端（5174）
- **作用**: 用一条命令启动完整的本地开发环境；若 `backend/.env` 不存在会自动从 `.env.example` 复制一份并提示填写 API Key。
- **操作**:
  ```bash
  cd /Users/douwen/Documents/HUAWEl/Shader-Agent/P2S-Agent
  ./start.sh start
  ```
- **预期效果**: 终端打印 `[P2S-Agent] Backend started (PID: ...)`、`[P2S-Agent] Frontend started (PID: ...)`，并以 `P2S-Agent started. Frontend: http://localhost:5174, Backend: http://localhost:8001` 收尾。根目录写出 `.backend.pid` 与 `.frontend.pid`。浏览器打开 `http://localhost:5174` 可见前端，`curl http://localhost:8001/health` 返回 `{"status":"ok","agent":"p2s-agent"}`。首次运行若无 `.env`，会先看到 `Warning: .env file not found. Copying from .env.example...`。
- **代码位置**: `start.sh`（`case "$1" in start)` 分支 → `check_env`/`start_backend`/`start_frontend`）；后端通过 `python3 -m uvicorn app.main:app --host 0.0.0.0 --port 8001 --reload` 启动（`start.sh:60`），前端通过 `npm run dev`（`start.sh:70`）。

#### 功能：停止服务
- **功能**: 关闭 `.pid` 文件记录的后端/前端进程
- **作用**: 终止由 `start` 拉起、并被记录到 `.backend.pid` / `.frontend.pid` 的进程。
- **操作**:
  ```bash
  ./start.sh stop
  ```
- **预期效果**: 打印 `Stopping services...` 与 `All services stopped`，并删除两个 `.pid` 文件。注意：`stop` 只杀 `.pid` 里记录的进程，捕获不到孤儿/重复的 vite 进程（这类需用 `clean`）。
- **代码位置**: `start.sh` 的 `stop_all()`（`start.sh:77`）。

#### 功能：强制清理（clean）
- **功能**: 强杀本项目所有残留进程（孤儿、重复 vite），按目录 + 端口精确定位
- **作用**: 清理 `stop` 抓不到的残留——包括属于本项目的 vite 进程及其 npm 父进程、签名匹配的 uvicorn 进程，以及占用 8001/5174 及 vite 自增端口 5175–5178 的进程。作用域限定本项目目录与端口，不会误杀同机其他项目（例如占用 5173 的 VFX-Agent）。
- **操作**:
  ```bash
  ./start.sh clean
  ```
- **预期效果**: 打印 `Cleaning up P2S-Agent processes (force)...` 与 `Cleanup complete. (Sibling projects / port 5173 untouched.)`；8001/5174/5175–5178 端口被释放，`.pid` 文件被删除。
- **代码位置**: `start.sh` 的 `clean()`（`start.sh:108`，含 `pgrep -f "$FRONTEND_DIR.*vite"`、`pkill -f "uvicorn app.main:app.*--port $BACKEND_PORT"`、端口循环 `5175 5176 5177 5178`）。

#### 功能：重启（restart）
- **功能**: 先 `clean` 再启动一套全新进程
- **作用**: 端口被占/进程错乱时一键恢复干净状态。
- **操作**:
  ```bash
  ./start.sh restart
  ```
- **预期效果**: 先执行清理输出，再依次启动后端、前端，最后打印 `P2S-Agent restarted. Frontend: http://localhost:5174, Backend: http://localhost:8001`。
- **代码位置**: `start.sh` 的 `restart()`（`start.sh:145`）。

#### 功能：查看状态（status）
- **功能**: 显示后端/前端运行状态
- **作用**: 读取 `.pid` 文件并用 `kill -0` 探测进程是否存活。
- **操作**:
  ```bash
  ./start.sh status
  ```
- **预期效果**: 打印 `=== P2S-Agent Status ===`，并分别显示 `Backend: Running (PID: ...)` / `Stopped`、`Frontend: Running (PID: ...)` / `Stopped`。
- **代码位置**: `start.sh` 的 `show_status()`（`start.sh:91`）。

#### 功能：安装依赖（install）
- **功能**: 安装后端（pip）+ 前端（npm）依赖
- **作用**: 首次部署时安装 Python 依赖（`backend/requirements.txt`）和前端依赖（`npm install`）。
- **操作**:

- **代码位置**: `start.sh` 的 `install_backend()`/`install_frontend()`（`start.sh:43`、`start.sh:50`）。

#### 功能：单独启动后端 / 前端
- **功能**: 只拉起后端（`backend`）或只拉起前端（`frontend`）
- **作用**: 调试时只跑其中一侧。
- **操作**:

- **预期效果**: `backend` 仅启动 uvicorn 并写 `.backend.pid`；`frontend` 仅启动 vite 并写 `.frontend.pid`。无参数或未知参数时打印用法 `Usage: ... {start|stop|restart|clean|status|install|backend|frontend}` 并以退出码 1 结束。
- **代码位置**: `start.sh` 的 `start_backend()`/`start_frontend()` 与 `case` 分支（`start.sh:178`–`start.sh:198`）。

#### 重要提示：Vite 端口自增坑
- **功能**: 前端端口可能不是 5174
- **作用**: Vite 在 `vite.config.ts` 中固定请求 `port: 5174`，但当 5174 被占用时 Vite 会自动顺延到 5175/5176/…。
- **操作**: 启动后查看前端日志的 `Local: http://localhost:51xx/`，以实际打印的端口为准访问。若想强制使用 5174，先 `./start.sh clean` 释放端口再启动。

- **代码位置**: `frontend/vite.config.ts`（`server.port = 5174`、`server.proxy` 中 `/api` 和 `/png-shader` 均 `target: http://localhost:8001`）；端口自增清理在 `start.sh:124`（`5175 5176 5177 5178`）。

### 二、环境变量配置（backend/.env）

> 启动时若无 `.env` 会自动从 `backend/.env.example` 复制。后端用两个 Settings 类分别读取同一个 `.env`：Web 层只取 host/port（`app/config.py`），Agent 层取 LLM/模型/截图/阈值等（`p2s_agent/config.py`）。两边都设了 `extra="ignore"`，因此互相多余的键不会导致启动报错。

#### 功能：LLM 默认（回退）模型配置
- **功能**: 候选生成/精修默认调用的单模型
- **作用**: 当请求未指定 preset、且无任何已配置 preset 时回退使用的模型；也是“legacy 单模型”默认值。
- **操作**: 编辑 `backend/.env`，按需填写（仅给出键名，不要写真实密钥）：
  ```
  LLM_API_KEY=<your key>
  LLM_BASE_URL=https://api.openai.com/v1
  LLM_MODEL=gpt-4o
  LLM_SUPPORTS_IMAGE=false
  ```
- **预期效果**: 重启后端后，未指定模型的运行会使用该配置。`LLM_SUPPORTS_IMAGE=true` 且 `LLM_MODEL=mimo-v2.5-pro` 时，实际解析的模型名会被改写为 `mimo-v2.5`。
- **代码位置**: `backend/p2s_agent/config.py` 的 `Settings.llm_*` 字段、`_default_llm()`、`_resolve_llm_model()`（`config.py:65`–`config.py:149`）。

#### 功能：预设模型槽位（前端可选，最多 3 个）
- **功能**: 前端模型选择器中可选的预设模型
- **作用**: 通过编号槽位 `MODEL_1_*` / `MODEL_2_*` / `MODEL_3_*` 定义最多 3 个前端可选模型。某槽位 `*_API_KEY` 留空即为“未配置”占位，前端中显示但禁用。**这些 API Key 永远不会被 `GET /api/models` 返回。**
- **操作**: 编辑 `backend/.env`（键名示例，密钥自填）：
  ```
  MODEL_1_ID=claude-opus
  MODEL_1_LABEL=Claude Opus
  MODEL_1_API_KEY=<your key>
  MODEL_1_BASE_URL=https://aihubmix.com/v1
  MODEL_1_MODEL=claude-opus-4-8
  MODEL_1_SUPPORTS_IMAGE=true
  # MODEL_2_*, MODEL_3_* 同理；API_KEY 留空即为占位
  ```
- **预期效果**: 重启后端后，`GET /api/models` 中出现对应 preset；填了 key 的 `configured: true`，未填的 `configured: false`。一个槽位只要 `MODEL_N_MODEL` 非空就会被收录（`MODEL_N_ID` 缺省时回退用 model 名，`MODEL_N_LABEL` 缺省时回退用 id）。
- **代码位置**: `backend/p2s_agent/config.py` 的 `model_N_*` 字段与 `model_presets` 属性（`config.py:72`–`config.py:176`）；样例见 `backend/.env.example`。

#### 功能：全局代理
- **功能**: 访问境外 API 的代理
- **作用**: 为所有 LLM 调用统一设置 HTTP/SOCKS5 代理。
- **操作**: 在 `backend/.env` 中（默认注释关闭）：
  ```
  PROXY=http://127.0.0.1:7890
  # 或 PROXY=socks5://127.0.0.1:7890
  ```
- **预期效果**: 重启后所有默认模型与 preset 模型（及自定义模型）的请求经此代理。
- **代码位置**: `backend/p2s_agent/config.py` 的 `proxy` 字段，注入到 `_default_llm()` 与 `model_presets`（`config.py:94`、`config.py:171`）；自定义模型在 `model_resolver.resolve_model_config()` 中也带上 `settings.proxy`。

#### 功能：服务地址（HOST/PORT/FRONTEND_URL）
- **功能**: 后端监听地址与前端 URL（CORS 白名单）
- **作用**: `HOST`/`PORT` 由 Web 层 `WebSettings` 读取；`FRONTEND_URL` 由 Agent 层 `Settings` 读取，并作为 CORS `allow_origins` 唯一白名单。
- **操作**: 在 `backend/.env`：
  ```
  HOST=0.0.0.0
  PORT=8001
  FRONTEND_URL=http://localhost:5174
  ```
  （注：`start.sh` 启动 uvicorn 时显式传入 `--port 8001`，会覆盖此处 `PORT`；`FRONTEND_URL` 仍影响 CORS。）
- **预期效果**: 浏览器从 `FRONTEND_URL` 源访问后端不被 CORS 拦截；若前端实际跑在自增端口并直连后端，源不匹配会被拦（走 vite 代理则无此问题）。
- **代码位置**: `backend/app/config.py` 的 `WebSettings.host/port`（`config.py:6`–`config.py:7`）；`backend/p2s_agent/config.py` 的 `frontend_url`（`config.py:97`）；CORS 注册在 `backend/app/main.py:30`（`allow_origins=[agent_settings.frontend_url]`）。

#### 功能：截图/渲染配置
- **功能**: 离屏渲染分辨率与超时
- **作用**: 控制 GLSL 渲染截图的宽高及单帧渲染超时（毫秒）。
- **操作**: 在 `backend/.env`：

  ```
  SCREENSHOT_WIDTH=512
  SCREENSHOT_HEIGHT=512
  ```

- **代码位置**: `backend/p2s_agent/config.py` 的 `screenshot_width`/`screenshot_height`/`render_timeout_ms`（`config.py:99`–`config.py:102`）。

#### 功能：保护区域否决阈值（PROTECT_VETO_SSIM_FLOOR/CEIL）
- **功能**: SSIM 强度 → 区域保护的否决阈值上下限
- **作用**: 由图编排逻辑消费，传入 `region_metrics.protect_region_threshold()`，根据 SSIM 强度决定是否否决对某区域的改动。
- **操作**: 在 `backend/.env` 追加（样例文件未列出，可手动添加；默认 floor=0.85、ceil=0.95）：
  ```
  PROTECT_VETO_SSIM_FLOOR=0.85
  PROTECT_VETO_SSIM_CEIL=0.95
  ```
- **预期效果**: 重启后区域保护否决的触发严格度随之变化（floor 越高→越容易判为需保护）。
- **代码位置**: `backend/p2s_agent/config.py` 的 `protect_veto_ssim_floor`/`protect_veto_ssim_ceil`（`config.py:106`–`config.py:107`）。

#### 功能：LangSmith 追踪（可选）
- **功能**: 开启 LangSmith 链路追踪
- **作用**: 默认关闭；开启后将运行轨迹上报 LangSmith。
- **操作**: 在 `backend/.env`：
  ```
  LANGSMITH_TRACING=true
  LANGSMITH_API_KEY=<your key>
  LANGSMITH_PROJECT=p2s-agent
  LANGSMITH_ENDPOINT=https://api.smith.langchain.com
  LANGSMITH_TAGS=p2s-agent
  ```
- **预期效果**: 重启后端，启动事件触发 `configure_langsmith()`；`LANGSMITH_TRACING=false`（默认）时不上报。
- **代码位置**: `backend/p2s_agent/config.py` 的 `langsmith_*` 字段（`config.py:110`–`config.py:114`）；启动钩子 `backend/app/main.py:122` 的 `configure_langsmith()`。

#### 功能：上传大小上限（MAX_UPLOAD_BYTES）
- **功能**: 多部分上传图片体积上限
- **作用**: 防止未鉴权客户端用超大图把 worker 撑爆内存。默认约 25 MB（`25 * 1024 * 1024`）。读取顺序：先在 `/run` 路由依赖里按 `Content-Length` 预检（413），再在解析后对真实字节复检。
- **操作**: 在 `backend/.env` 追加（直接读环境变量，不在样例中；按字节数填写）：
  ```
  MAX_UPLOAD_BYTES=26214400
  ```
  验证（构造一个超限请求会被拒）：
  ```bash
  curl -i -X POST http://localhost:8001/png-shader/run \
    -H "Content-Length: 999999999" \
    -F "image=@/path/to/image.png"
  ```
- **预期效果**: 超过上限时返回 HTTP 413，body `{"detail":"upload exceeds maximum size of <N> bytes"}`。非图片 content-type 返回 415；字节无法解码为图片返回 422。仅允许 `image/png`、`image/jpeg`、`image/jpg`、`image/webp`。
- **代码位置**: `backend/app/api/guards.py`（`_MAX_UPLOAD_BYTES`、`_ALLOWED_IMAGE_CONTENT_TYPES`、`_guard_upload`、`_check_content_length`）；在 `/run` 上挂为依赖见 `backend/app/api/routers/core.py:58`。

#### 功能：并发运行上限（MAX_ACTIVE_RUNS）
- **功能**: 顶层 pipeline worker 的全局并发背压
- **作用**: 每个顶层运行（`/run`、`/branch-refine`、fusion）提交时须占用一个全局信号量槽位；满了则返回 429 而非无限开线程。默认 6。变体子运行单独用 `_MAX_VARIANT_CONCURRENCY=2` 限流，不占全局槽位。
- **操作**: 在 `backend/.env` 追加（直接读环境变量）：
  ```
  MAX_ACTIVE_RUNS=6
  ```
  验证：连续快速提交多于上限个 `/png-shader/run` 任务，观察是否出现 429。

#### 功能：自定义模型 base_url 白名单（LLM_BASE_URL_ALLOWLIST）
- **功能**: 放行被 SSRF 防护默认拦截的内网/回环地址
- **作用**: 当请求体携带“自定义模型”时，其 `base_url` 会被 SSRF 校验（只允许 http/https，且拒绝 loopback/link-local/私网/`169.254.169.254` 元数据 IP/`localhost`）。本白名单（逗号分隔）可对指定 host 或解析出的 IP 放行，例如自建本地 LLM。


- **预期效果**: 在白名单内的 host/IP 即使是内网/回环也允许作为自定义模型 base_url；不在白名单的内网地址在 `/run` 解析自定义模型时返回 422，detail 形如 `custom model base_url host '...' is a blocked (internal/metadata) address`。

### 五、Serverless CLI 一次性运行（无需 FastAPI）

#### 功能：单张 PNG → GLSL 一次性转换（命令行）
- **功能**: 不启动 Web 服务，直接在命令行把一张 PNG 转成 GLSL
- **作用**: 默认走确定性 / AI-off 模式（`llm_enabled=False`、`vlm_judge_enabled=0`、`refinement_mode="off"`、`glsl_render_enabled=False`、`cv_enabled=True`），因此**无需任何 LLM/VLM 密钥**即可运行。只从 `p2s_agent.*` 导入，不依赖 `app.*` 或 fastapi。
- **操作**:
  ```bash
  # 在 backend 目录运行，确保 p2s_agent 可被导入
  cd /Users/douwen/Documents/HUAWEl/Shader-Agent/P2S-Agent/backend
  python3 -m p2s_agent.cli \
    --image /absolute/path/to/image.png \
    --out /absolute/path/to/cli_out \
    --log-level INFO
  # 可选：基于种子 shader 做精修而非全新生成
  python3 -m p2s_agent.cli --image /abs/img.png --seed-glsl /abs/seed.glsl --out /abs/cli_out
  ```
  参数：`--image PATH`（必填，源 PNG）；`--seed-glsl PATH`（可选，给定则精修该 shader）；`--out DIR`（输出目录，默认 `./cli_out`）；`--log-level {DEBUG,INFO,WARNING,ERROR}`（默认 WARNING）。
- **预期效果**: 退出码 0；输出目录写出两个文件：`selected_shader.glsl`（选中的 GLSL，无结果时为空文件）与 `metrics.json`（含 `run_id`、`final_score`、`objective_metrics`、`quality_router`、`selected_candidate_id`）。stdout 打印四行：
  ```
  run_id:      <id>
  final_score: <0.xxxx 或 n/a>
  glsl:        <.../selected_shader.glsl>
  metrics:     <.../metrics.json>
  ```
  错误情形：`--image` 文件不存在 → stderr `error: image not found: <path>`，退出码 1；`--seed-glsl` 不存在 → stderr `error: seed GLSL not found: <path>`，退出码 1；pipeline 抛异常 → stderr `error: pipeline failed: <msg>`，退出码 1。
- **代码位置**: `backend/p2s_agent/cli.py` 的 `main()`（`argparse` 定义见 `cli.py:41`–`cli.py:70`；AI-off 的 `build_input_spec(...)` 见 `cli.py:96`；调用 `run_png_shader_pipeline(image_path, input_spec, seed_glsl=...)` 见 `cli.py:111`；产物写出见 `cli.py:126`–`cli.py:149`）；依赖 `p2s_agent.core.pipeline.graph.run_png_shader_pipeline` 与 `p2s_agent.core.pipeline.input_spec.build_input_spec`。

- **操作**:
  ```bash
  curl -X POST http://localhost:8001/png-shader/runs/run_ab12cd34/branch-refine \
    -H "Content-Type: application/json" \
    -d '{
      "checkpoint_id":"final:selected",
      "mode":"refine",
      "feedback":"保留构图，把光晕做得更柔和",
      "locks":{},
      "stop_parent":false,
      "quality":{"max_refinement_iterations":2},
      "constraints":{"use_preferences":true,"regions":[]}
    }'
  ```
  字段：`checkpoint_id`（默认 `final:selected`）、`mode`（须属于 `MODES`，如 `refine`/`polish`/`explore` 等）、`feedback`（按 mode 校验）、`locks`（对象）、`quality`（对象）、`stop_parent`（布尔）、`constraints`（可选结构化约束）。

- **代码位置**: `branch.py` → `branch_refine`（`POST /png-shader/runs/{run_id}/branch-refine`）。

  ```bash
      "base_run_id":"run_base0001",

    }'
  ```
  字段：`base_run_id`（必填）、`source_run_ids`（可选）、`feedback`（可选）、`regions`（每个含 `source_run_id`，会并入源集合）。

# 第三部分 · Studio 工作台（单次运行的创作与检视）

## 工作台（Studio）页面 — 单次运行的创作与检视

> 适用范围：单次运行的「上传 → 选模型/策略 → 生成 → 观察候选/时间线 → 检视着色器/评分/对比/模型 I/O → 调参/精细控制/DSL 层 → 预览」全流程。
> 前提：后端运行在 `http://localhost:8001`，前端开发服务器在 `http://localhost:5174`。Studio 是默认页面（URL 不带 `?view=canvas` 时即进入）。
> 注意：Studio 内的所有面板都从 `PngShaderContext` 读取同一份运行状态（见 `frontend/src/context/PngShaderContext.tsx` 与 `frontend/src/components/PngShaderView.tsx`）。下文「UI」操作均在 Studio 页内进行；「API」操作给出可直接复制的 `curl`。

### 一、上传与启动

#### 功能：上传 PNG（拖拽或点击）
- **作用**：选择要转换为着色器的输入图片，作为整条流水线的参考图。
- **操作**：
  1. 打开 `http://localhost:5174`（默认即 Studio）。
  2. 把一张 `.png` 拖入顶部虚线框「拖入 PNG 或点击上传」区域；或单击该区域，在系统文件选择器中选 `.png`。
- **预期效果**：上传框变为「已选文件」状态，显示文件名（等宽字体）、一个 ×（清除）按钮和一个绿色「运行」按钮。拖拽悬停时边框变绿并发光。非 PNG 文件被忽略（`handleFile` 仅接受 `image/png` 或以 `.png` 结尾的文件）。
- **代码位置**：`frontend/src/components/PngShaderView.tsx`（`handleFile` / `handleDrop` / `handleInputChange` / 上传区 JSX）。

#### 功能：清除已选文件
- **作用**：撤销当前选择的图片，回到空上传态。
- **操作**：在已选文件行点击 × 按钮。
- **预期效果**：文件名清空，上传框恢复为「拖入 PNG 或点击上传」提示；隐藏的 file input 也被重置。
- **代码位置**：`PngShaderView.tsx`（`handleClear`，`fileInputRef.current.value = ""`）。

#### 功能：从已有着色器开始（Seed GLSL）
- **作用**：以一段已有 GLSL 作为闭环起点（而非从零生成），可粘贴文本或上传 `.glsl/.frag/.txt` 文件。
- **操作**：
  1. 在「从已有着色器开始 / Seed GLSL」卡片勾选复选框。
  2. 在出现的多行文本框粘贴 GLSL（Shadertoy `mainImage` 或普通 `main` 片元着色器），或点击下方文件选择器上传着色器文件（文件内容会读入文本框）。
- **预期效果**：勾选后文本框与文件选择器出现；运行时该 GLSL 作为 `seed_glsl` 一并提交。运行中（loading）时这些控件被禁用。
- **代码位置**：`PngShaderView.tsx`（`seedEnabled` / `seedGlsl` / `handleSeedFile`，运行时 `onRun(selectedFile, seedEnabled ? seedGlsl : undefined)`）；提交字段见 `frontend/src/hooks/usePngShader.ts`（`formData.append("seed_glsl", seedGlsl)`）；后端 `backend/app/api/routers/core.py` `run_png_shader` 的 `seed_glsl: Optional[str] = Form(...)`。

#### 功能：运行流水线（生成着色器）
- **作用**：把图片、AI 模式、模型选择与策略打包提交后端，启动一次 PNG→着色器的完整生成。
- **操作（UI）**：选好文件后点击绿色「运行」按钮。
- **操作（API，等价请求）**：
  ```bash
  curl -X POST http://localhost:8001/png-shader/run \
    -H "x-request-id: run-demo-1" \
    -F "image=@/path/to/input.png;type=image/png" \
    -F 'input_spec_json={"llm":{"mode":"auto"},"quality":{"refinement_mode":"auto","max_refinement_iterations":3},"model":{"base_url":"https://api.example.com/v1","api_key":"sk-...","model":"qwen-vl-plus","supports_image":true}}'
  ```
  （`seed_glsl` 可选，追加 `-F "seed_glsl=void mainImage(out vec4 o, in vec2 u){ o = vec4(1.0); }"`。）
- **预期效果**：按钮变为「运行中...」并带旋转 loader；返回 HTTP 200 + JSON，含 `run_id`、`status`（如 `queued`/`running`/`done`）。前端随后每 1 秒轮询 `GET /png-shader/status/{run_id}` 直到终态，下方各分析面板逐步填充。
- **代码位置**：`PngShaderView.tsx`（`handleRun`）；`usePngShader.ts`（`onRun` 构造 FormData、`POST /png-shader/run`、`pollStatus` 轮询 `GET /png-shader/status/{id}`）；后端 `backend/app/api/routers/core.py`（`@router.post("/run")`、`@router.get("/status/{run_id}")`）。

### 四、运行结果检视与编辑（下排四栏）

#### 功能：图像对比（Image Diff）— 参考图 vs 着色器实时渲染
- **作用**：左侧显示参考图，右侧用 WebGL 实时渲染当前活动 GLSL（256×256），便于肉眼对比。
- **操作**：运行后自动渲染；切换候选/迭代/编辑参数时右侧渲染随之更新。
- **预期效果**：右侧显示着色器动态渲染；当显示的是预览/编辑版本时，标题旁出现绿色「预览: <标签>」徽章（编辑态标签为 `edited`）。着色器编译失败时覆盖红色「着色器错误」+错误摘要。整个面板复用单个 WebGL 上下文避免上下文泄漏。
- **代码位置**：`frontend/src/components/ImageDiffPanel.tsx`（`ShaderRenderer` + `toShaderToyFragment`）；活动 GLSL 选择优先级见 `PngShaderView.tsx`（`activeGlsl = workingGlsl ?? previewGlsl ?? result.selected_glsl`）。

#### 功能：参数面板（Parameters）— 可视化调参
- **作用**：解析当前 GLSL 中的 `#define` 可调参数，按分组（Background / Layer N / Colors / 其他）渲染滑块、颜色拾取器（vec3 颜色、`*_r/_g/_b` 三元组）、vec2/vec4 分量滑块；调节即改写 GLSL。
- **操作**：
  1. 在「Parameters」面板拖动标量滑块或在数值框输入；点击颜色块用拾色器改色；展开/折叠分组标题。
  2. 点击右上「重置」（RotateCcw）图标恢复到本结果的原始 GLSL。
- **预期效果**：每次改动即时改写 GLSL，并以 `edited` 标签反映到「图像对比」右侧渲染；改动按「活动结果键」缓存（切到别的候选再切回会保留各自编辑）。无可调参数时显示「No tunable parameters found」；无着色器时显示「No shader selected」。
- **代码位置**：`frontend/src/components/PngShaderParamPanel.tsx`（`parseShader`/`updateShaderParam`、`extractGroups`、各 Control 组件）；编辑回写见 `PngShaderView.tsx`（`handleParamGlslChange` 写入 `workingGlslByKey[activeKey]`）。

#### 功能：已选候选（Selected Candidate）— GLSL 预览
- **作用**：显示当前活动候选的来源徽章、候选 ID、字符数和 GLSL 源码预览（默认前 30 行，可展开全文）。
- **操作**：查看下排第三栏「已选候选」；若行数超过 30，点击「展开更多（N 行）」/「收起」。
- **预期效果**：显示 GLSL 文本（等宽）；AI 来源显示「AI 生成」徽章；未选候选时显示「未选择候选」。
- **代码位置**：`frontend/src/components/DslLayerPanel.tsx`。

#### 功能：补全可调参数（参数化 GLSL）
- **作用**：调用模型把着色器里的硬编码常量提升为可调 `#define` 参数，使「参数面板」能暴露更多旋钮。
- **操作（UI）**：在「已选候选」面板右上点击「补全可调参数」（Sliders 图标）。
- **操作（API）**（需已有 run_id；提交当前 GLSL）：
  ```bash
  curl -X POST http://localhost:8001/png-shader/parameterize/<RUN_ID> \
    -F "glsl=void mainImage(out vec4 o, in vec2 u){ o = vec4(0.8,0.2,0.1,1.0); }"
  ```
- **预期效果**：按钮转圈（parameterizing）；成功后返回含 `glsl`、`param_count_before`、`param_count_after` 的 JSON，新 GLSL 写入当前活动结果，「参数面板」出现更多可调项。失败时上传区下方显示「补全可调参数失败：…」。

# 第五部分 · 核心管线能力（理解「预期效果」的原理层）

## 核心管线能力（Core Pipeline）验证手册

> 适用对象：希望从算法层面验证 PNG → GLSL/Shadertoy 着色器生成质量的人工测试者。
> 后端默认地址 `http://localhost:8001`，前端开发服务器 `http://localhost:5174`。
> 本节解释每个 API/UI 操作背后"预期效果"的算法含义（候选生成 → 评分 → 选择 → 后处理优化/精修 → 质量门）。

### 一、端到端生成流程（一次完整运行）

#### 功能：完整 PNG→Shader 流水线（LangGraph 四节点 + 后处理）
- **功能**: 一次生成运行（preprocess → candidates → scoring → selection → post-pipeline）
- **作用**: 上传一张 PNG，流水线先做图像预处理提取结构特征，再生成多个候选着色器，逐个渲染打分并经质量路由产出 `final_score`，选出最佳候选，最后按需做参数优化/修订/残差补层/LLM 闭环精修/VLM 终评，返回结构化结果。
- **操作**:
  ```bash
  curl -X POST http://localhost:8001/png-shader/run \
    -F "image=@/path/to/icon.png"
  ```
  仅文件即用全部默认值（AI 关闭、CV 开启、`refinement_mode=auto`）。要观察更多阶段可附 `input_spec_json`：
  ```bash
  curl -X POST http://localhost:8001/png-shader/run \
    -F "image=@/path/to/icon.png" \
    -F 'input_spec_json={"candidates":{"cv_enabled":true},"quality":{"mode":"balanced","max_iterations":5,"max_added_layers":4,"refinement_mode":"off"}}'
  ```
  返回 `run_id` 后轮询状态：
  ```bash
  curl http://localhost:8001/png-shader/status/run_xxxxxxxx
  ```
- **预期效果**: `/run` 返回 HTTP 200，body 含 `{"run_id","status":"running",...}`。轮询 `/status/{run_id}` 直到 `status:"completed"`，此时 body 含 `selected_candidate_id`、`selected_dsl`（DSL 候选时）或 `selected_glsl`、`objective_metrics`、`quality_router`（内含 `final_score`/`quality_band`/`next_action`）、`scoreboard`、`refinement_summary`。同时磁盘 run 目录写出 `manifest.json`、`reference_input.png`、`input_spec.json`、`preprocess.json`、`candidates.json`、`scoreboard.json`、`objective_metrics.json`、`quality_router.json`、`selected_shader.glsl` 等工件。
- **代码位置**: 入口 `backend/p2s_agent/core/pipeline/graph.py::run_png_shader_pipeline`（LangGraph 编排 `_build_graph` / 节点 `node_preprocess`/`node_candidates`/`node_scoring`/`node_selection`，后处理 `_run_post_pipeline`）；HTTP 入口 `backend/app/api/routers/core.py::run_png_shader`（`POST /png-shader/run`），状态 `get_status`（`GET /png-shader/status/{run_id}`）。

#### 功能：无服务器一次性 CLI（确定性 / AI-off）
- **功能**: `p2s_agent.cli` 命令行单次转换
- **作用**: 不启动 FastAPI、不需任何 LLM/VLM 密钥，确定性地把一张 PNG 转成 GLSL，并写出选中的着色器与指标 JSON。用于离线验证核心管线本身（无 AI 干扰）。
- **操作**:
  ```bash
  cd /Users/douwen/Documents/HUAWEl/Shader-Agent/P2S-Agent/backend
  python3 -m p2s_agent.cli --image /path/to/icon.png --out ./cli_out --log-level INFO
  ```
  可选 `--seed-glsl /path/to/seed.glsl` 走 seed 精修入口（见下节）。
- **预期效果**: 退出码 0；stdout 打印 `run_id` / `final_score` / `glsl` 路径 / `metrics` 路径；`./cli_out/selected_shader.glsl` 为非空着色器，`./cli_out/metrics.json` 含 `final_score`、`objective_metrics`、`quality_router`、`selected_candidate_id`。CLI 强制 `llm_enabled=False`、`vlm_judge_enabled=0`、`refinement_mode="off"`、`max_refinement_iterations=0`，所以结果完全由 baseline/rule/decompose/CV 候选决定。
- **代码位置**: `backend/p2s_agent/cli.py::main`（构造 `build_input_spec(... candidates={llm_enabled:False, cv_enabled:True}, quality={vlm_judge_enabled:0, refinement_mode:"off"}) `后调 `run_png_shader_pipeline`）。

### 二、候选生成与图像预处理

#### 功能：图像预处理 / 特征提取
- **功能**: `preprocess_image`（结构特征抽取）
- **作用**: 用 Pillow 从输入 PNG 提取尺寸、是否含透明、`alpha_coverage`（非透明像素占比）、`palette`（16 级量化的 top-5 颜色）、`color_count_estimate`（32 级 bin 计数）、`edge_sharpness`（相邻像素差>20 的比例）、`component_count_estimate`（32×32 下采样上的 BFS 连通域数）、`texture_score`（8×8 块亮度方差归一化）、以及派生的 `photo_like_score` / `gradient_score`。这些特征驱动 rule/decompose/CV 候选与修订补丁的几何/配色决策。超过 1024px 的图会先按比例缩放再做逐像素分析（防 DoS）。
- **操作**: 对任意 `run_id` 运行后取磁盘工件，或直接读 `/status` 的 `preprocess` 字段：
  ```bash
  curl -s http://localhost:8001/png-shader/status/run_xxxxxxxx | python3 -c "import sys,json;print(json.load(sys.stdin)['preprocess'])"
  ```
- **预期效果**: 得到上述键的字典；`alpha_coverage` 在 [0,1]（不透明图为 1.0），`palette` 恰好 5 个 `#RRGGBB`，`edge_sharpness`/`texture_score`/`gradient_score`/`photo_like_score` 均在 [0,1]。run 目录还写出 `preprocess.json`、`normalized_input.png`（128×128）、`alpha_mask.png`、`llm_reference_input.png`（对透明图合成到黑底）。
- **代码位置**: `backend/p2s_agent/core/pipeline/preprocess.py::preprocess_image` / `save_preprocess_artifacts`；流水线节点 `graph.py::node_preprocess`。

#### 功能：候选池（baseline / rule / decompose / CV / LLM / fallback）
- **功能**: `run_candidate_pool`（多来源候选生成）
- **作用**: 并行生成多种候选着色器：`baseline`（确定性基础）、`rule`（规则化 DSL）、`decompose`（需 OpenCV：量化+连通域+图元拟合，几何从像素"测量"而非全局统计猜测）、`cv`（CV 特征驱动）、`llm`（可选，DSL 或 raw GLSL）、`fallback`（兜底，保证总有可编译候选）。每个候选完成 DSL 校验+编译，或对 raw GLSL 做静态校验。
- **操作**: 默认开启 CV（`candidates.cv_enabled=true`）。要看 decompose 候选需安装 OpenCV。要加入 LLM 候选：`input_spec_json` 中 `"candidates":{"llm_enabled":true,"llm_implementation":"auto"}`（需配置模型，见 `/models`）。运行后查看候选板：
  ```bash
  curl -s http://localhost:8001/png-shader/status/run_xxxxxxxx | python3 -c "import sys,json;print(json.load(sys.stdin)['scoreboard'])"
  ```
- **预期效果**: `scoreboard.candidates` 列出多行，每行带 `source`（baseline/rule/decompose/cv/llm/fallback）、`final_score`、`quality_band`、`compile_success`。`candidates.json` 保存全部候选细节；若某来源失败会在该行 reason / 错误中体现，但 fallback 保证至少一个可编译候选。
- **代码位置**: `backend/p2s_agent/core/pipeline/pool.py::run_candidate_pool`；decompose 算法 `backend/p2s_agent/core/pipeline/decompose.py::decompose_to_dsl` / `fit_primitive_layer`（`DECOMPOSE_AVAILABLE` 取决于 cv2）。

#### 功能：候选选择（含 VLM 近分仲裁）
- **功能**: `select_best_candidate` + VLM 近分仲裁
- **作用**: 在所有 `compile_success` 候选中，先按 `final_score` 降序、再按 `priority`（越小越优）、再按 GLSL 长度（越长=特征越多）选最佳；无可编译候选时退回 fallback。当 `glsl_render_enabled` 且最佳 GLSL 候选分数不低于最佳 DSL，则优先输出 GLSL。若 `vlm_judge_enabled`，且 Top-1/Top-2 分差 < `vlm_tie_epsilon`（默认 0.05），调 `judge_pairwise` 让视觉模型在近分时仲裁（仅调整排序用分数，不改客观 `quality_router.final_score`）。
- **操作**: 默认即生效。要触发近分仲裁需 `quality.vlm_judge_enabled=1` 且 `candidates.llm_enabled=true`（VLM 才被启用）。
- **预期效果**: `selected_candidate_id` 指向得分最高（或近分仲裁后胜出）的候选；该候选 `scoreboard` 行 `selected:true`。近分仲裁时被选候选 reason 含 `vlm pairwise judge ...` 文案。
- **代码位置**: `pool.py::select_best_candidate`；`graph.py::node_selection`（`judge_pairwise` 调用、`vlm_tie_epsilon` 比较）。

### 三、GLSL 模式 vs DSL 模式

#### 功能：DSL 模式（可栅格化场景）
- **功能**: DSL 候选编译 + 离线栅格渲染
- **作用**: DSL 候选用 `compile_dsl` 编译成 GLSL，并用纯 Python/PIL 渲染器 `render_dsl_to_image` 离线栅格化打分（不依赖浏览器），因此可在无 WebGL 环境下评估。DSL 模式支持坐标下降参数优化、修订补丁、残差补层这些只能在结构化场景上做的操作。
- **操作**: 默认走 DSL（`target.backend=glsl` 但候选多为 DSL）。`candidates.glsl_render_enabled=false`（默认）时 raw GLSL 候选无法栅格化打分会被记 0 分，从而倾向选 DSL。
- **预期效果**: 选中候选 `selected_dsl` 非空，`objective_metrics` 含 v2 键（`rmse`/`mask_iou`/`edge_iou`/`grid_color_sim`/`simple_ssim`）；run 目录写 `selected_dsl.json`。
- **代码位置**: `backend/p2s_agent/core/pipeline/scoring.py::_evaluate_dsl`（`render_dsl_to_image` + `compute_objective_metrics` + `route`）；编译 `backend/p2s_agent/core/dsl/compiler.py::compile_dsl`。

#### 功能：GLSL 模式（raw Shadertoy GLSL + WebGL 评分）
- **功能**: raw GLSL 候选的浏览器 WebGL 渲染评分
- **作用**: 当启用 `glsl_render_enabled`（或 LLM 以 `shadertoy_glsl`/`auto` 实现自动开启）时，raw GLSL 候选通过 Playwright 无头 Chromium 注入预览页渲染并截图，再与参考图比对打分。GLSL 模式支持 `#define` 参数优化与 GLSL 闭环精修，但不支持 DSL 专属的残差补层/修订补丁。
- **操作**:
  ```bash
  curl -X POST http://localhost:8001/png-shader/run \
    -F "image=@/path/to/icon.png" \
    -F 'input_spec_json={"candidates":{"llm_enabled":true,"llm_implementation":"shadertoy_glsl","glsl_render_enabled":true}}'
  ```
  注意：需要安装 Playwright 浏览器，且前端预览页可达（`frontend_url`，默认 `http://localhost:5174`）。
- **预期效果**: 选中候选 `selected_glsl` 非空、`selected_dsl` 为 null；`objective_metrics` 含 `render_backend:"webgl"`、`backend_rasterized:true`。若 WebGL 渲染失败，该候选记 0 分并在 reason 标注 `webgl scoring failed`。
- **代码位置**: `scoring.py::_evaluate_glsl_with_webgl` / `_make_render_glsl_fn`；渲染服务 `backend/p2s_agent/core/render/browser_render.py::render_multiple_frames` / `render_and_screenshot`（等待 `window.__shaderReady`，超时 `render_timeout_ms`）。

#### 功能：Seed-GLSL 入口（精修外部着色器）
- **功能**: 以外部 GLSL 为种子直接精修（跳过候选生成）
- **作用**: 提供 `seed_glsl` 时，流水线不生成候选，而是把种子适配为可渲染的 Shadertoy GLSL（`adapt_seed_glsl`），合成单个 GLSL 候选并初评，随后进入相同的后处理（强制 `glsl_render_enabled=True`、`llm_enabled=True`、`refinement_mode` 默认 `on`）。人在环路的 branch-refine 也用此入口承接父检查点的着色器。
- **操作**:
  ```bash
  curl -X POST http://localhost:8001/png-shader/run \
    -F "image=@/path/to/icon.png" \
    -F "seed_glsl=void mainImage(out vec4 o, in vec2 f){ o=vec4(f/iResolution.xy,0.0,1.0); }"
  ```
- **预期效果**: 200 + `run_id`；run 目录写 `seed_input.glsl` 与 `seed_adapted.glsl`。若种子无法适配为可渲染 GLSL，运行被标记失败（`ValueError`）。`refinement_mode` 默认按 seed 走 `on`，因此通常会看到至少一次 GLSL 精修迭代。
- **代码位置**: `graph.py::_run_seed_glsl_path` / `run_png_shader_pipeline(seed_glsl=...)`；适配 `backend/p2s_agent/core/pipeline/seed_glsl.py::adapt_seed_glsl`/`build_seed_candidate`；HTTP 透传 `core.py::run_png_shader`（`seed_glsl` Form 字段，默认置 `refinement_mode="on"`）。

### 四、质量 / 评分路由与 final_score

#### 功能：客观指标计算
- **功能**: `compute_objective_metrics`（v2 位置感知指标）
- **作用**: 把参考图与渲染图各自 alpha 合成到背景色后，计算 `mse`、`rmse`、`simple_ssim`、`mask_iou`（前景掩码 IoU）、`edge_iou`（边缘 IoU）、`grid_color_sim`（8×8 网格 Lab ΔE 相似度）、以及 `nonblank_render`（非空白）、`within_shader_budget`（字符数 ≤ `max_shader_chars`）。v1 旧键（`color_histogram_score`/`alpha_coverage_diff`/`edge_density_diff`）也保留兼容。
- **操作**: 读 `/status` 的 `objective_metrics` 字段或 run 目录 `objective_metrics.json`。
- **预期效果**: 字典含上述键；位置感知指标 `mask_iou`/`edge_iou`/`grid_color_sim` 存在即说明走 v2 评分公式。
- **代码位置**: `backend/p2s_agent/core/metrics/compute.py::compute_objective_metrics`（`_mask_iou_arrays`/`_edge_iou_arrays`/`_grid_color_similarity`）。

#### 功能：质量路由器与 final_score 计算
- **功能**: `quality_router.route` + `compute_final_score`
- **作用**: 先过硬门（编译/渲染/预算/非空白），任一不过直接 `status:"failed"`、`final_score:0.0`、`next_action:"fallback"`。全过后用加权公式算 `final_score`（v2：SSIM×0.30 + grid_color_sim×0.25 + mask_iou×0.20 + edge_iou×0.15 + (1−rmse)×0.10；有语义分时再按 objective×0.7 + 语义均值×0.3 融合），并据分数分档与建议下一步：≥0.85 excellent/final，≥0.70 good/final，≥0.55 acceptable/optimize，≥0.40 poor/revise，<0.40 failed/fallback。
- **操作**: 读 `/status` 的 `quality_router`：
  ```bash
  curl -s http://localhost:8001/png-shader/status/run_xxxxxxxx | python3 -c "import sys,json;q=json.load(sys.stdin)['quality_router'];print(q['final_score'],q['quality_band'],q['next_action'])"
  ```
- **预期效果**: `quality_router` 含 `status`/`quality_band`/`next_action`/`final_score`(∈[0,1])/`failure_type`/`reason`/`protected_aspects`。`next_action` 决定后处理是否触发优化/修订（见第五节）。
- **代码位置**: `backend/p2s_agent/core/metrics/quality_router.py::route` / `compute_final_score`（阈值 0.85/0.70/0.55/0.40）。

### 五、两条精修闭环与检查点间的变化

#### 功能：DSL 后处理优化 / 修订 / 残差补层
- **功能**: 坐标下降参数优化 + 修订补丁（带回滚）+ 残差补层
- **作用**: 选中候选为 DSL 且 `next_action∈{optimize,revise}` 且 `max_iterations>0` 时，用坐标下降优化层参数（仅当提分才接受）；`next_action∈{revise,fallback}` 时按 `failure_type` 构造修订补丁（如 color 失败→把层填充改成参考调色板首色；几何失败→重置 center/radius/size 到源覆盖率），失败自动回滚；当 `max_added_layers>0` 且分数 < `refinement_high_score_stop` 时基于残差自动新增图层。
- **操作**:
  ```bash
  curl -X POST http://localhost:8001/png-shader/run \
    -F "image=@/path/to/icon.png" \
    -F 'input_spec_json={"quality":{"max_iterations":8,"max_added_layers":4,"refinement_mode":"off"}}'
  ```
- **预期效果**: run 目录出现 `optimization/optimizer.json`、`revision/revision.json`、`residual_layers/residual.json`（按触发情况）。`/status` 顶层 `optimization`/`revision` 含 `improved`、`initial_score`→`final_score`。被接受的改动会令选中候选 `final_score` 升高、`selected_dsl` 更新。
- **代码位置**: `graph.py::_run_post_pipeline`（DSL 分支）；优化 `backend/p2s_agent/core/pipeline/optimizer.py::optimize_candidate`、修订 `revision.py::apply_revision_with_rollback` + `refinement.py::_build_revision_patch`、补层 `residual_layers.py::add_residual_layers`；接受 `scoring.py::_accept_improvement`。

#### 功能：DSL LLM 精修闭环
- **功能**: `run_dsl_refinement_loop`（LLM 迭代修订 DSL）
- **作用**: 每次迭代把当前 DSL 渲染图、客观指标、网格色差报告（`grid_color_report`）、（可选）VLM rubric 语义提示、近几轮历史摘要喂给 LLM，让它产出修订 DSL；校验→编译→渲染→打分；只接受提分迭代（rollback 会把回退说明作为下一轮反馈）。早停条件：`best_score≥high_score_stop` 或 `≥threshold`、连续无提升达 `patience`、近 3 轮极差<0.02 的 stagnation。运行中可读取实时策略更新与 `stop_requested`。
- **操作**:
  ```bash
  curl -X POST http://localhost:8001/png-shader/run \
    -F "image=@/path/to/icon.png" \
    -F 'input_spec_json={"candidates":{"llm_enabled":true},"quality":{"refinement_mode":"on","max_refinement_iterations":3,"refinement_threshold":0.80,"refinement_high_score_stop":0.92,"refinement_min_improvement":0.01,"refinement_patience":2}}'
  ```
  （需配置 LLM 模型）查看每轮检查点：`GET /png-shader/runs/{run_id}/checkpoints`。
- **预期效果**: run 目录出现 `refinement/iter_1.json`、`iter_2.json`…（每轮记 `score_before/score_after/delta/improved/changes_summary/accepted`）。`/status` 的 `refinement_summary` 含 `enabled/decision/iterations/initial_score/final_score/improved/stop_reason`，`refinement_history` 为逐轮列表；`/checkpoints` 列出 `refinement_iter` 检查点。两个相邻 iter 的差异可由 `changes_summary`（如 `layer.fill.color: #aaa → #bbb`）看出"检查点间变化"。
- **代码位置**: `backend/p2s_agent/core/pipeline/refinement.py::run_dsl_refinement_loop`（门控 `_should_run_refinement`）；编排在 `graph.py::_run_post_pipeline`。

#### 功能：GLSL LLM 精修闭环
- **功能**: `run_glsl_refinement_loop`（LLM 迭代修订 raw GLSL，带"全新重写"机制）
- **作用**: 与 DSL 闭环同构，但作用于 raw GLSL：静态校验失败/WebGL 渲染失败会把具体错误作为下一轮反馈；增量停滞时触发一次 `fresh restart`，要求 LLM"换一种技术从头重写"（`max_fresh_restarts=1`）。差异摘要按改动行/`#define` 给出。
- **操作**:
  ```bash
  curl -X POST http://localhost:8001/png-shader/run \
    -F "image=@/path/to/icon.png" \
    -F 'input_spec_json={"candidates":{"llm_enabled":true,"llm_implementation":"shadertoy_glsl","glsl_render_enabled":true},"quality":{"refinement_mode":"on","max_refinement_iterations":3}}'
  ```
- **预期效果**: run 目录出现 `glsl_refinement/iter_*.json`；`refinement_summary.kind=="glsl"`。若某轮 GLSL 渲染失败，该 iter 记 `error_type:"render_failed"`、`score_after:0.0` 且不被接受。触发重写时对应 iter 含 `fresh_start:true`。
- **代码位置**: `backend/p2s_agent/core/pipeline/glsl_refinement.py::run_glsl_refinement_loop`（`_trigger_fresh_restart` / `_build_failure_log_note`）；编排在 `graph.py::_run_post_pipeline`（GLSL 分支，`_glsl_refinement_evaluate`）。

#### 功能：精修门控与"为什么没精修"
- **功能**: `_should_run_refinement`（是否进入闭环）
- **作用**: `refinement_mode=off` 永不精修；`auto` 仅当分数 < `threshold` 时精修（且若 `auto` 但 LLM 未启用→不精修，decision=`auto_llm_disabled`）；`on` 始终精修（除非 ≥`high_score_stop`）；`max_refinement_iterations<=0` → 不精修，decision=`max_refinement_iterations_zero`。人在环路 branch 用 `force_first` 强制至少一轮。注意：精修门用的是"真客观分"（`quality_router.final_score`），VLM 近分仲裁的排序加成会被剥除，避免低分候选被加成顶过阈值而漏精修。
- **操作**: 读 `refinement_summary.decision` 字段判断未精修原因。
- **预期效果**: `refinement_summary.enabled` + `decision` 给出明确原因，如 `auto_threshold_reached`/`auto_llm_disabled`/`max_refinement_iterations_zero`/`refinement_mode_off`/`force_enabled`。
- **代码位置**: `refinement.py::_should_run_refinement`；门控分值 `scoring.py::_gate_quality_score`；编排 `graph.py::_run_post_pipeline`。

### 六、保护区域硬否决（constraint_score）

#### 功能：protect-region 硬否决
- **功能**: `evaluate_protect_veto` / `compute_region_metrics`（保护区 SSIM 否决）
- **作用**: 人在环路 branch-refine 提交 `mode:"protect"` 的矩形区域后，精修每一轮把候选渲染图与"约束设定时的基线渲染"（选中检查点的外观，固定锚点）在各保护区裁剪比对 SSIM。若任一保护区 SSIM < 其强度阈值（强度 0..1 线性映射到 `[protect_veto_ssim_floor, protect_veto_ssim_ceil]`，默认 0.85→0.95，强度 0.5→0.90），该轮被硬否决：不接受、计入 patience、并给 LLM 一条 `[PROTECT VIOLATION]` 反馈要求别动保护区。`constraint_score` = 各保护区 SSIM 均值（无保护区时默认 1.0）。
- **操作**: 先 `/run` 得到父 `run_id`，再创建带保护区的定向子运行：
  ```bash
  curl -X POST http://localhost:8001/png-shader/runs/run_PARENT/branch-refine \
    -H "Content-Type: application/json" \
    -d '{
      "checkpoint_id":"final:selected",
      "mode":"refine",
      "feedback":"让背景更亮，但不要改动中心图标",
      "constraints":{
        "regions":[{"id":"r1","label":"center icon","mode":"protect",
          "geometry_type":"rect","geometry":{"x":0.35,"y":0.35,"w":0.30,"h":0.30},
          "strength":0.6}]
      }
    }'
  ```
  阈值可调环境变量 `PROTECT_VETO_SSIM_FLOOR` / `PROTECT_VETO_SSIM_CEIL`。
- **预期效果**: 子运行返回 200 + 子 `run_id`；run 目录写 `constraints.json`、`directed_acceptance.json`。`refinement_summary.protect_regions` 含 `{count, veto_count, last_constraint_score}`；被否决的 iter 记 `rejected_reason:"protect_region_veto"`、`region_veto:[{id,label,ssim,threshold,violated}]`、`constraint_score`。即使整体分提高，只要保护区退化就不会被接受。
- **代码位置**: `backend/p2s_agent/core/pipeline/region_metrics.py::evaluate_protect_veto` / `compute_region_metrics` / `protect_region_threshold`；基线锚点 `graph.py::_build_region_veto_fn`；阈值配置 `backend/p2s_agent/config.py::Settings.protect_veto_ssim_floor/ceil`；HTTP 透传 `backend/app/api/routers/branch.py::branch_refine`（`protect_regions` 注入 `pipeline_extra`）；约束模型 `backend/p2s_agent/orchestration/human_constraints.py`。

#### 功能：定向接受（允许小幅降分以达成人类目标）
- **功能**: directed acceptance（目标感知 VLM 仲裁）
- **作用**: branch-refine 的 `refine`/`polish` 模式下，若某轮分数小幅下降但仍在容差内（`refine` 容差 0.03，`polish` 为 0），且目标感知 VLM 判定候选更符合用户反馈（裁决 "B"），则破例接受该轮（记 `human_goal_override:"accepted_score_drop"`，不计入 patience）。需 VLM 可用，否则退化为纯指标接受。
- **操作**: 同上 branch-refine（`mode:"refine"` 自动启用 `vlm_judge_enabled=1` 与 directed acceptance）。
- **预期效果**: `directed_acceptance.json` 含 `{enabled, feedback, mode, score_drop_tolerance, require_vlm_for_score_drop}`；被破例接受的 iter 在 `refinement_history` 中带 `human_goal_override`。
- **代码位置**: `branch.py::branch_refine`（构造 `directed_acceptance`）；接受逻辑 `refinement.py::run_dsl_refinement_loop` 与 `glsl_refinement.py::run_glsl_refinement_loop`（`directed_pairwise_judge` 分支）。

### 七、VLM 终评门（语义融合）

#### 功能：VLM 终评 rubric 融合
- **功能**: `judge_rubric` 终评门
- **作用**: 当 `vlm_judge_enabled` 且选中候选有渲染图时，用视觉模型对最终渲染打 rubric 语义分，并按 `objective×0.7 + 语义均值×0.3` 融合回 `final_score`，同时记录 `failure_type`、`differences`、`revision_hints`。
- **操作**: `quality.vlm_judge_enabled=1` 且 `candidates.llm_enabled=true`（启用模型）。
- **预期效果**: run 目录写 `judge/final_rubric.json`；`/status` 的 `vlm_judge` 含 `objective_score`、`blended_score`、`semantic_scores`、`failure_type`；选中候选 `final_score` 被替换为 `blended_score`，`quality_router` 增 `semantic_scores`/`vlm_failure_type` 字段。
- **代码位置**: `graph.py::_run_post_pipeline`（"VLM final gate" 段，`judge_rubric` + `compute_final_score(metrics, semantic_scores)`）；判官 `backend/p2s_agent/core/llm/vlm_judge.py`。

### 八、运行期控制（精修过程中可观察/可干预）

#### 功能：实时策略更新（运行中调阈值）
- **功能**: `PATCH /runs/{run_id}/strategy`
- **作用**: 运行中修改质量策略字段（如 `refinement_threshold`/`refinement_high_score_stop`/`refinement_min_improvement`/`refinement_patience`/`max_refinement_iterations`），精修循环每轮开头通过 `strategy_reader` 读到并即时生效；调低 `max_refinement_iterations` 到当前轮以下可提前停。
- **操作**:
  ```bash
  curl -X PATCH http://localhost:8001/png-shader/runs/run_xxxxxxxx/strategy \
    -H "Content-Type: application/json" \
    -d '{"quality":{"refinement_threshold":0.7,"max_refinement_iterations":1}}'
  ```
- **预期效果**: 200 + `{"strategy":{...合并后...},"strategy_revision":N}`（递增）；后续 iter 反映新阈值，可能提前出现 `stop_reason:"user_lowered_cap"`。非 running 状态返回 409。
- **代码位置**: `core.py::patch_strategy`；循环消费 `refinement.py`/`glsl_refinement.py` 的 `strategy_reader` 段。

#### 功能：停止运行
- **功能**: `POST /runs/{run_id}/stop`
- **作用**: 置 `stop_requested`，精修循环下一轮开头检测到即停（`stop_reason:"user_stop"`），不中断当前已在跑的迭代。
- **操作**:
  ```bash
  curl -X POST http://localhost:8001/png-shader/runs/run_xxxxxxxx/stop
  ```
- **预期效果**: 200 + `{"stopping":true}`；`refinement_summary.stop_reason` 最终为 `user_stop`。非 running/queued 返回 409。
- **代码位置**: `core.py::stop_run`；循环检测 `refinement.py`/`glsl_refinement.py`（`live.get("stop_requested")`）。

> 算法层备注：精修每一轮的 GLSL 都会写入 `iter_*.json.compile_glsl`，前端可逐迭代预览；运行中流水线通过 `publish_partial` 先发"基线快照"（候选板+初选），再每轮发增量（best_score/best_glsl/history），因此轮询 `/status` 可看到 `refinement_history` 逐步增长——这是"检查点间变化"在 UI 上的实时来源。对应 `graph.py::_run_post_pipeline` 的 `_publish_iteration` 与 baseline 发布段。
